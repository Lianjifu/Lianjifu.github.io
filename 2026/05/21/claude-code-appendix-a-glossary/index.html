附录 A：术语表

思考维度 内容
引导思考 展开
它为什么存在
  • 25 章技术深潜塞了 65 个术语——中文名、英文名、首次出现的章节，一次性查清楚
  • 既是词典也是索引：想看某个概念在书里的分析，跟着首见章节跳转就行
它解决什么问题
  • 新读者面对一堆缩写（XAA、DCE、CIMD、MCP、PTL）时的茫然感
  • 跨章节追踪术语定义时的翻阅需求
它在系统中的位置
  • 作为附录不参与主线叙事，但看书过程中随时可能需要翻过来查
  • A-Z 字母序排列方便快速定位
它如何工作
  • 每个术语包含英文名、中文译名、一句话定义和首见章节
  • A-Z 分组表格组织，一目了然
它如何实现
  • 纯 Markdown 表格，兼容 GitBook 渲染
  • 跟正文中的术语使用保持同步
不同平台如何做
  • 传统技术书：通常把术语表放书末或每章开头——本书延续了这个惯例
  • 代码注释生成：有些项目用 TypeDoc 自动生成 API 术语表，但本书的手工维护更可控
优势是什么？
  • 优势
    • 一站式 65 个术语查阅，不用跨章节搜
    • 首见章节索引方便回溯
它解决什么问题
  • 新读者面对大量自创缩写（XAA、DCE、CIMD、MCP、PTL）和专有概念时的茫然感
  • 跨章节追踪术语定义时的高频查阅需求
它在系统中的位置
  • 作为附录不参与主线叙事
  • 按字母顺序排列的结构方便快速查找
它如何工作
  • 每个术语包含英文名、中文译名、一句话定义和首见章节
  • 表格形式组织，A-Z 字母分组
它如何实现
  • 纯 Markdown 表格，与 GitBook 渲染引擎兼容
  • 手动维护，与正文中术语的使用保持同步
不同平台如何做
  • 传统技术文档：通常将术语表置于书末或每章开头，Claude Code 源码分析延续了这一惯例
  • 代码注释（JSDoc）：有些项目使用 TypeScript 注释生成 API 术语表（TypeDoc），与本附录互补而非替代
优缺点是什么
  • 优势
    • 一站式查阅 65 个术语，避免跨章节搜索
    • 首见章节索引帮助读者按需回溯阅读
