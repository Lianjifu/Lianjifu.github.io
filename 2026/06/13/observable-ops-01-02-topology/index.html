业务 02 · 拓扑建模

智能系统运维可观测性 · 业务拓扑与依赖关系建模

1. 痛点问题

拓扑建模是智能运维的基石，但当前运维体系在拓扑感知、拓扑维护、拓扑应用三个环节都存在显著痛点。

1.1 拓扑盲区：看不见的依赖才是最大的风险

在大型分布式系统中，服务之间存在复杂的依赖关系。传统的监控体系只能看到单点指标（如 CPU、内存），无法呈现服务间的拓扑连接和依赖层级。这导致：

痛点场景 现状描述 后果
级联故障扩散 服务 A 故障，不知道有多少下游服务受影响 故障范围无法评估
依赖脆弱点 核心服务依赖某个不稳定中间件，无法识别 单点故障导致全瘫
变更影响面 发布前不知道会影响到哪些服务 变更风险不可控
容量黑洞 某个基础服务容量不足，上层服务全部受影响 资源规划失效

级联故障传播示意：

flowchart LR CACHE[缓存服务 故障点] --> A[订单服务] CACHE --> B[支付服务] CACHE --> C[推荐服务] A --> D[用户端] A --> E[商家端] B --> D C --> F[运营端] style CACHE fill:#ff6b6b style D fill:#ffccbc style E fill:#ffccbc style F fill:#ffccbc

真实案例： 某电商平台因为一个基础缓存服务故障，导致订单服务、支付服务、推荐服务全部级联崩溃，影响范围远超预期，损失超过 200 万。

1.2 拓扑信息的碎片化与过时

当前运维中，拓扑信息散落在多个系统，且难以保持时效性：

4 大系统各自盲区：

系统 知道什么 不知道什么

1.3 拓扑建模的三大难题

flowchart LR subgraph 难题["拓扑建模三大难题"] D1[动态性 实例随时扩缩容] D2[异构性 多种架构混部] D3[规模性 万级实例/十万级依赖] end subgraph 影响["后果"] E1[快照生成时 已过时] E2[统一建模 困难] E3[采集存储 成本高] end D1 --> E1 D2 --> E2 D3 --> E3 style 难题 fill:#fff3e0 style 影响 fill:#fce4ec

难题详解：

难题 描述 影响
动态性 服务实例随时扩缩容，容器漂移，IP 变化 拓扑快照在生成时已过时
异构性 传统微服务、K8s Pod、Serverless、遗留系统混部 统一建模困难
规模性 万级服务实例，十万级依赖关系 全量采集和存储成本高

2.2 分层目标

拓扑建模按抽象层次分为 3 层，自底向上逐层抽象：

flowchart TB L1[L1 业务拓扑层 业务域 → 服务集群] L2[L2 服务拓扑层 服务 → 服务调用] L3[L3 资源拓扑层 服务 → 基础设施] L3 --> L2 --> L1 style L1 fill:#e1bee7 style L2 fill:#fff9c4 style L3 fill:#bbdefb

L1：业务拓扑层

  • 目标：建立业务域到技术实现的映射
  • 内容：
    • 业务域 → 产品线 → 服务集群
    • 用户旅程 → 关键路径 → 服务依赖
    • SLA 级别 → 核心服务 → 降级策略
  • 价值：让运维理解业务，让业务感知运维

L2：服务拓扑层

  • 目标：建立服务间调用关系的准确拓扑图
  • 内容：
    • 服务 → 服务 调用拓扑
    • API → Backend → Database 链路
    • Frontend → BFF → Domain Service 调用链
  • 价值：故障定位、变更评估、依赖分析

L3：资源拓扑层

  • 目标：建立服务与基础设施的承载关系
  • 内容：
    • 服务 → Pod/容器 → Node 主机
    • 服务 → 负载均衡器 → 网络
    • 服务 → 存储卷 → 磁盘
  • 价值：容量规划、故障定位、资源治理
